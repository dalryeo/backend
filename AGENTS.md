# AGENTS.md

## 프로젝트 전제
- 달려 백엔드 API는 곧 런칭할 Java/Spring Boot 운영 서비스이다.
- 데이터베이스 기준은 PostgreSQL이며, 배포 기준은 실제 workflow와 Dockerfile을 우선 확인한다.
- 예상 초기 규모는 DAU 수백~수천 수준이므로 점진적 안정화를 우선한다.

## 코드 변경 원칙
- Controller는 얇게 유지하고 비즈니스 규칙은 Service에 둔다.
- 도메인 검증은 Service 또는 전용 validator에서 보호한다.
- 공통 응답은 `CommonResponse`, 예외 응답은 `common.exception` 구조를 우선 재사용한다.
- `record` 변경은 저장 검증, `record/outbox`, `weekly_user_stats` 주간 집계와 랭킹 영향을 함께 확인한다.
- 작업 범위와 무관한 대규모 리팩터링은 하지 않는다.

## 테스트와 검증
- 동작이 바뀌면 변경된 요구사항을 검증하는 테스트를 추가하거나 수정한다.
- 단위 테스트는 Service, validator, 계산 로직처럼 좁은 규칙을 빠르게 검증할 때 우선한다.
- 통합 테스트는 API 계약, 예외 응답, 영속성, 트랜잭션, outbox 영향을 검증할 때 사용한다.
- 날짜/시간 변경은 `Asia/Seoul`, `OffsetDateTime`, offset 포함 여부를 명시적으로 확인한다.
- 완료 전에는 변경 범위와 직접 연결된 테스트를 먼저 실행하고, 머지 전에는 가능한 한 `./gradlew test`로 전체 회귀를 확인한다.
- 보고할 때는 실행한 검증과 실행하지 못한 검증을 구분한다.

## 브랜치/커밋/PR
- 브랜치는 `main` 중심의 짧은 생명주기 전략을 따른다.
- 브랜치명은 `docs/shared/conventions/깃-브랜치-전략.md`를 따른다.
- 커밋 규칙은 `docs/shared/conventions/깃-커밋-컨벤션.md`를 따른다.
- PR 본문은 `.github/PULL_REQUEST_TEMPLATE.md` 형식을 따른다.
- 브랜치명이 작업 문서나 대시보드에 명시돼 있으면 별도 컨벤션 문서를 읽지 않는다.
- 커밋 규칙도 마찬가지로, 요청 범위에 커밋이 포함되지 않으면 컨벤션 문서를 읽지 않는다.


## 문서와 계약
- 문서와 코드가 다르면 `build.gradle`, `src/main/resources/application.yml`, 테스트 코드, workflow, Dockerfile 같은 실행 기준 파일을 우선 확인한다.
- 요청/응답 필드, 상태 코드, 에러 응답 형식은 외부 API 계약으로 취급한다.
- API 계약이 바뀌면 구현, 테스트, 문서를 같은 변경 범위에서 맞춘다.

## 협업 방식
- 주요 기술 결정은 변경 이유, 대안, 운영 영향을 짧게 설명한다.
- 선택지가 여러 개이면 권장안을 먼저 제시하고, 사용자의 결정이 필요한 지점만 질문한다.
- 코드 리뷰 의견은 그대로 반영하지 않고 프로젝트 구조, 운영 영향, 변경 비용을 기준으로 반영/보류/분리 여부를 제안한다.
- 사용자가 개념 이해를 요청하면 중학생 수준으로 쉬운 비유와 단계적 설명을 우선한다.
- 사용자가 리뷰 피드백을 함께 학습하려는 경우, 구현 전에 문제의 본질과 판단 기준을 먼저 설명한다.
- 스킬을 설정했을 때 md 파일을 한 번 읽으면 그 세션에는 다시 읽지 않는다. 다시 읽어야 한다면 사용자에게 어떤 이유로 인해 읽어야 하는지 안내한다.

## 불확실할 때
- 운영 의도나 영향 범위를 로컬에서 확인할 수 없으면 추측하지 않고 가정과 위험을 먼저 보고한다.
- 변경 범위가 예상보다 커지면 구현을 계속하기보다 범위를 먼저 정리한다.
- API 계약, DB 스키마, 배포 설정 변경은 영향 범위를 확인한 뒤 진행한다.

## 하지 말 것
- 작업 범위와 무관한 대규모 리팩터링
- 근거 없는 API 계약 변경
- 비밀값 하드코딩
- 운영 의도를 모르는 상태에서 모니터링, Sentry, 배포 설정 제거
- 요청 범위에 명시되지 않은 연관 문서를 맥락 파악 목적으로 탐색하는 것
- 요청 범위 밖의 문서를 수정하는 것
- 이슈 상태 확인이 요청에 포함되지 않은 경우 gh issue list 등 외부 API 호출
- 문서 상태 갱신처럼 단순한 작업에 완료 후 재검증 루프를 실행하는 것

## 완료 보고
- 파일을 생성하거나 수정한 경우에는 전체 내용을 다시 출력하지 않고, 변경한 파일과 완료 여부만 간단히 보고한다.
- 사용자가 요청하지 않으면 생성한 문서나 코드의 전체 본문을 응답에 붙이지 않는다.
- 검증을 실행한 경우에는 실행한 명령과 결과만 간단히 보고한다.
