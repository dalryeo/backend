# 러닝 기록 저장 시 도메인 검증이 없어 데이터 신뢰도가 흔들릴 수 있던 문제

- Status: Archived
- Audience: Engineers, Codex
- Source of Truth: No
- Last Reviewed: 2026-06-20

## 왜 이 작업을 하게 되었는가

달려에서 러닝 기록은 단순 저장 데이터가 아닙니다.
하나의 세션 기록이 아래 기능들의 공통 기준이 됩니다.

- 기록 상세 조회
- 주간 요약
- 티어 계산
- 랭킹 집계
- 추후 확장될 성장 분석 기능

즉 기록 저장 시점의 품질이 보장되지 않으면,
이후 기능들이 모두 같은 오염된 데이터를 기준으로 계산하게 됩니다.

현재 구조에서는 요청 DTO에서 필수값 여부와 양수 여부 정도만 확인하고 있어,
"값이 들어왔는가"는 검증하지만
"실제 러닝 기록으로 성립하는가"는 충분히 검증하지 못하고 있었습니다.

## 실제로 어떤 문제가 생길 수 있었는가

기존 저장 구조에서는 아래와 같은 비정상 기록이 들어와도 저장될 여지가 있었습니다.

- `endAt`이 `startAt`보다 빠른 기록
- `durationSec`와 실제 `endAt - startAt`이 맞지 않는 기록
- `distanceKm`, `durationSec`, `avgPaceSecPerKm`가 서로 정합성이 맞지 않는 기록
- 지나치게 비현실적인 페이스, 심박수, 칼로리 값
- 서버 기준 현재 시각보다 미래에 생성된 기록

이런 값이 한번 저장되면 문제가 한 API에만 머무르지 않습니다.

- 주간 거리 합계가 실제보다 부풀려질 수 있습니다.
- 평균 페이스가 왜곡될 수 있습니다.
- 티어 점수가 비정상적으로 높거나 낮아질 수 있습니다.
- 랭킹 신뢰도가 떨어질 수 있습니다.

결국 기록 저장은 "입력 API"가 아니라
"서비스 전체의 기준 데이터를 통과시키는 관문"으로 봐야 했습니다.

## 원인 분석

핵심 원인은 입력값 검증과 도메인 검증이 분리되지 않았다는 점이었습니다.

기존 검증은 주로 아래 수준에 머물러 있었습니다.

- 필수값인지
- 양수인지
- 문자열 형식이 맞는지

하지만 러닝 기록은 각 필드가 서로 관계를 갖는 데이터입니다.

- 시작 시간과 종료 시간은 순서가 맞아야 합니다.
- 거리, 시간, 페이스는 서로 계산상 맞아야 합니다.
- 값 범위도 "양수"만으로는 충분하지 않습니다.

즉 개별 필드 validation만으로는 서비스가 기대하는 기록 품질을 보장할 수 없었고,
서비스 레이어에서 도메인 규칙을 기준으로 한 정합성 검증이 필요했습니다.

## 어떻게 해결하려고 하는가

해결 방향은 단순합니다.
비정상 기록은 저장 단계에서 차단하고 DB에 들어가지 못하게 합니다.

이번 작업에서는 아래 기준을 중심으로 검증 로직을 보강할 계획입니다.

- `endAt`은 반드시 `startAt`보다 뒤여야 합니다.
- `durationSec`는 실제 시간 차이와 허용 오차 범위 내에서 일치해야 합니다.
- `avgPaceSecPerKm`는 `distanceKm`, `durationSec`으로 계산한 값과 허용 오차 범위 내에서 일치해야 합니다.
- 미래 시각 기준 기록은 저장하지 않습니다.
- 비현실적인 `distanceKm`, `durationSec`, `avgPaceSecPerKm`, `avgHeartRate`, `caloriesKcal`는 저장하지 않습니다.
- 검증 실패 시 클라이언트가 처리 가능한 400 에러와 명확한 메시지를 반환합니다.

핵심은 "나중에 집계할 때 걸러내는 것"이 아니라
"애초에 공통 기준 데이터로 사용할 수 없는 기록은 저장하지 않는 것"입니다.

## 구현 시 같이 정리할 부분

기록 검증을 추가할 때는 단순 if 문 몇 개를 넣는 수준으로 끝내지 않으려고 합니다.
같이 정리해야 할 포인트는 아래와 같습니다.

- 기록 저장용 도메인 검증 규칙을 서비스 레이어 한 곳에 모읍니다.
- 검증 실패 사유를 일관된 예외 형태로 정리합니다.
- 컨트롤러마다 흩어진 에러 응답 계약과 충돌하지 않도록 공통 처리 방식을 맞춥니다.
- 단위 테스트와 통합 테스트에 비정상 입력 케이스를 추가합니다.

이렇게 해야 이후 중복 기록 방지, 스냅샷 집계, 랭킹 최적화 같은 작업을 올릴 때도
"어떤 기록이 유효한가"라는 전제가 흔들리지 않습니다.

## 구현 1단계에서 먼저 한 일

가장 먼저 한 작업은 DTO 레벨의 기본 범위 검증을 보강하는 것이었습니다.

이 단계에서는 필드 간 관계를 보지 않고,
"이 값 하나만 봐도 너무 이상하다"는 입력을 먼저 걸러내는 데 집중했습니다.

적용한 기준은 아래와 같습니다.

- `distanceKm`: `0.1km` 이상, `100km` 이하
- `durationSec`: `60초` 이상, `12시간` 이하
- `avgPaceSecPerKm`: `120초/km` 이상, `3600초/km` 이하
- `avgHeartRate`: 앱 단독 기록에서는 센서 값이 없을 수 있어 `0` 또는 `30~240`
- `caloriesKcal`: 앱 단독 기록에서는 추정값이 없을 수 있어 `0` 또는 `1~10000`

이 값들은 정교한 운동 분석 기준이라기보다,
정상 사용자 기록은 거의 막지 않으면서 명백한 이상값을 차단하기 위한 1차 방어선으로 잡았습니다.

즉 이 단계의 목적은 완성된 검증이 아니라,
서비스 레이어까지 도달할 필요가 없는 비정상 입력을 가장 앞단에서 거르는 것이었습니다.

## 구현 2단계에서 한 일

두 번째 단계에서는 검증 규칙과 예외를 서비스 안에 흩뿌리지 않기 위해
기록 전용 validator와 예외 구조를 분리했습니다.

이 단계에서 만든 구성은 아래와 같습니다.

- `RecordValidationErrorCode`
- `RecordValidationException`
- `RunningRecordValidator`

이 validator에는 아래 정합성 규칙을 모았습니다.

- `endAt`은 반드시 `startAt`보다 뒤여야 합니다.
- 실제 시간 차이와 `durationSec` 차이는 `±5초` 이내여야 합니다.
- 계산 페이스와 `avgPaceSecPerKm` 차이는 `±15초/km` 이내여야 합니다.
- 미래 시각 기록은 현재 시각 기준 `+5분`까지만 허용합니다.

이 단계의 핵심은 아직 저장 로직을 바꾸는 것이 아니라,
"기록 정합성 규칙을 한 군데에 모으고, 실패 사유를 코드로 구분할 수 있게 만든 것"입니다.

이 구조를 먼저 만들어 두어야 이후 저장 서비스, 글로벌 예외 처리, 테스트가
같은 기준을 재사용할 수 있습니다.

## 구현 3단계에서 한 일

세 번째 단계에서는 분리해 둔 validator를 실제 저장 흐름에 연결했습니다.

즉 검증 규칙이 단순히 존재만 하는 상태에서,
`saveRecord()`가 저장 전에 반드시 검증을 거치도록 바꿨습니다.

이 변경 이후의 저장 흐름은 아래와 같습니다.

1. DTO 기본 validation 수행
2. 서비스에서 `RunningRecordValidator` 실행
3. 검증 통과 시에만 `RunningRecord` 생성 및 저장

이 단계의 의미는 단순합니다.
이제부터는 비정상 기록이 "나중에 발견되는 문제"가 아니라
"아예 저장되지 않는 문제"로 바뀌었습니다.

즉 시간 역전, 지속시간 불일치, 평균 페이스 불일치, 미래 시각 초과 같은 입력은
DB에 남기지 않고 저장 단계에서 차단하게 됐습니다.

## 구현 4단계에서 한 일

네 번째 단계에서는 검증 실패를 클라이언트가 이해할 수 있는 응답 형태로 정리했습니다.

이전 구조에서는 400, 401 같은 에러 상태여도
응답 본문이 `success=true`로 내려가고 있었습니다.

이 구조는 HTTP 상태 코드를 보지 않는 클라이언트나,
성공/실패를 공통 응답 객체 기준으로 해석하는 화면에서 혼선을 만들 수 있습니다.

그래서 아래와 같이 정리했습니다.

- `CommonResponse.failure(...)` 추가
- `RecordValidationException` 전용 400 처리 추가
- `MethodArgumentNotValidException` 전용 400 처리 추가
- 기존 `AuthException`, `IllegalArgumentException`도 `success=false`로 반환하도록 정리

특히 요청 DTO 검증 실패의 경우에는
첫 번째 대표 메시지와 함께 필드별 오류를 `errors`에 담아 내려주도록 맞췄습니다.

이 단계 이후에는 기록 저장 실패가 아래처럼 더 명확해집니다.

- HTTP status: `400 Bad Request`
- 본문 `success`: `false`
- 본문 `data.code`: 에러 코드
- 본문 `data.message`: 사용자/클라이언트가 이해할 수 있는 메시지

즉 저장 로직에서 막는 것과,
왜 막혔는지를 응답으로 설명하는 것이 이제 같은 흐름으로 연결되었습니다.

## 구현 5단계에서 한 일

다섯 번째 단계에서는 서비스 레벨 검증이 실제로 기대한 대로 동작하는지 확인하는 단위 테스트를 추가했습니다.

이 단계에서 검증한 케이스는 아래와 같습니다.

- 정상 기록은 저장되는지
- `endAt <= startAt`인 경우 저장이 거부되는지
- `durationSec`와 실제 시간 차이가 허용 오차를 넘으면 거부되는지
- `avgPaceSecPerKm`가 거리와 시간 계산 결과와 맞지 않으면 거부되는지
- 미래 시각 기록이 거부되는지
- 허용 오차 경계값에서는 정상 통과하는지

특히 마지막 경계값 테스트를 넣은 이유는 중요합니다.
검증 로직은 틀린 값을 막는 것도 중요하지만,
플랫폼 반올림 차이처럼 허용해야 하는 정상 오차까지 막지 않는 것도 중요하기 때문입니다.

즉 이 단계는 단순히 "예외가 나는지"를 보는 테스트가 아니라,
우리가 정한 검증 정책이 너무 느슨하지도, 너무 빡빡하지도 않은지 확인하는 단계였습니다.

## 구현 6단계에서 한 일

여섯 번째 단계에서는 단위 테스트를 넘어,
실제 API 요청 기준으로 실패 응답 계약이 유지되는지 확인하는 통합 테스트를 추가했습니다.

이번 단계에서 확인한 케이스는 두 종류입니다.

- DTO 범위 검증 실패
- 서비스 도메인 검증 실패

DTO 범위 검증 실패 테스트에서는
거리, 시간, 페이스, 심박수, 칼로리를 모두 비정상 값으로 보내고 아래를 확인했습니다.

- HTTP status가 `400`인지
- 본문 `success`가 `false`인지
- `data.code`가 `BAD_REQUEST`인지
- `data.errors`에 필드별 메시지가 담기는지
- DB에 실제 레코드가 추가되지 않았는지

도메인 검증 실패 테스트에서는
`startAt`과 `endAt`을 같게 보내서 시간 역전 검증을 유도하고 아래를 확인했습니다.

- HTTP status가 `400`인지
- 본문 `success`가 `false`인지
- `data.code`가 기록 전용 에러 코드인지
- 메시지가 도메인 규칙에 맞게 내려오는지
- DB에 레코드가 저장되지 않았는지

즉 이 단계에서는
"서비스 내부에서 예외가 발생한다"는 것만 보는 대신,
"클라이언트가 실제로 어떤 응답을 받는가"와
"실패한 요청이 DB를 오염시키지 않는가"까지 같이 검증하게 되었습니다.

## 구현 7단계에서 확인한 내용

일곱 번째 단계에서는 관련 테스트를 실제로 실행했습니다.

먼저 `RecordServiceTest` 단위 테스트를 분리 실행했을 때는 통과했습니다.
즉 현재 추가한 DTO 범위 검증, 도메인 validator, 저장 전 차단 로직 자체는
서비스 레벨 기준으로 기대한 대로 동작하고 있음을 확인했습니다.

반면 `ApiContractIntegrationTest`는 애플리케이션 컨텍스트 기동 단계에서 실패했습니다.

실패 원인은 현재 코드 변경이 아니라 테스트 실행 환경의 datasource 설정이었습니다.
실제 실패 메시지는 아래와 같았습니다.

- `Failed to determine suitable jdbc url`
- `Failed to configure a DataSource: 'url' attribute is not specified`

즉 통합 테스트는 `application-test.yml`이 참조하는
`TEST_DB_URL` 또는 `DB_URL`이 현재 실행 환경에 없어 시작되지 못한 상태였습니다.

이 단계에서 확인한 핵심은 두 가지입니다.

- 서비스 레벨 검증 로직 자체는 테스트로 통과했다.
- 통합 테스트 전체 실패는 현재 구현 결함이 아니라 로컬 테스트 DB 환경 미설정 때문이다.

따라서 이후 조치는 검증 로직 수정이 아니라,
통합 테스트를 돌릴 수 있는 test datasource 환경을 맞추는 방향으로 가져가야 합니다.

## 구현 8단계에서 한 일

여덟 번째 단계에서는 통합 테스트가 실제로 돌아갈 수 있도록
로컬 PostgreSQL에 전용 테스트 DB를 준비하고 테스트를 다시 실행했습니다.

이때 확인한 점은 다음과 같습니다.

- 로컬 PostgreSQL 서버 자체는 실행 중이었다.
- 문제는 테스트 환경변수가 비어 있어서 datasource URL을 결정하지 못한 것이었다.
- 기존 `postgres` 데이터베이스를 그대로 쓰면 테스트의 `deleteAll()` 때문에 실제 데이터가 지워질 위험이 있었다.

그래서 아래 순서로 진행했습니다.

1. `dalryeo_test` 전용 데이터베이스 생성
2. `schema-init.sql`로 테이블과 티어 메타데이터 초기화
3. `TEST_DB_URL`, `TEST_DB_USERNAME`, `TEST_DB_PASSWORD`를 지정해 관련 테스트 재실행

이후 결과는 다음과 같았습니다.

- `RecordServiceTest` 통과
- `ApiContractIntegrationTest` 통과

즉 이번 작업으로 추가한 DTO 범위 검증, 도메인 validator, 저장 전 차단 로직,
실패 응답 계약, 서비스 단위 테스트, API 통합 테스트까지 모두 실제 실행 기준으로 검증되었습니다.

## 이 작업에서 특히 조심할 점

검증을 강화할 때 가장 조심해야 하는 부분은
"실제 사용자 기록도 과하게 막아버리는 것"입니다.

예를 들어 모바일 플랫폼별 수집 오차나 반올림 차이 때문에
`durationSec`, `paceSecPerKm`가 완벽히 일치하지 않을 수 있습니다.

그래서 이번 작업은 아래 원칙으로 가져가려고 합니다.

- 명백히 잘못된 값은 차단합니다.
- 플랫폼 오차로 볼 수 있는 작은 차이는 허용 오차 범위 안에서 수용합니다.
- 허용 오차는 코드에 하드코딩한 매직 넘버가 아니라 의미 있는 상수로 둡니다.
- 테스트도 경계값 중심으로 작성합니다.

## 왜 서버가 값을 보정해서 저장하지 않고 거부하려고 하는가

처음에는 서버가 값을 조금 고쳐서 저장하는 방법도 생각할 수 있습니다.
예를 들어 `durationSec`, `distanceKm`, `avgPaceSecPerKm`가 서로 맞지 않으면
서버가 계산해서 하나를 다시 맞춰 저장하는 방식입니다.

하지만 현재 단계에서는 이 방법이 더 위험하다고 판단했습니다.

이유는 서버가 어떤 값이 진짜 원본인지 확실히 알 수 없기 때문입니다.

예를 들어 아래처럼 값이 서로 충돌하는 기록이 들어올 수 있습니다.

- 거리: 5km
- 시간: 20분
- 평균 페이스: 6분/km

이 경우 셋 중 하나 이상은 잘못된 값입니다.
문제는 서버가 아래 중 무엇을 기준으로 삼아야 하는지 알 수 없다는 점입니다.

- 시간이 틀린 것인지
- 페이스가 틀린 것인지
- 거리가 틀린 것인지
- 일시정지 구간 때문에 계산 방식이 다른 것인지

현재 요청 구조와 DB 스키마에는 이런 판단을 뒷받침할 원본 메타데이터가 없습니다.
즉 서버가 조용히 값을 고쳐 저장하면,
정확한 정제가 아니라 추측에 기반한 데이터 변형이 됩니다.

이렇게 되면 아래 문제가 생깁니다.

- 나중에 어떤 값이 원래 사용자 기록이었는지 추적하기 어려워집니다.
- 랭킹, 티어, 분석 계산이 서버가 임의로 바꾼 값을 기준으로 동작하게 됩니다.
- 클라이언트나 수집 파이프라인의 실제 오류가 서버에서 가려집니다.
- 사용자 문의가 왔을 때 왜 특정 값으로 저장됐는지 설명하기 어려워집니다.

그래서 이번 작업에서는 아래 원칙을 택하려고 합니다.

- 작은 반올림 오차는 허용 오차 범위 안에서 수용합니다.
- 서로 충돌하는 값은 서버가 임의로 고치지 않습니다.
- 무엇이 맞는지 확실하지 않으면 저장하지 않고 거부합니다.

즉 지금 단계의 목표는
"서버가 데이터를 똑똑하게 보정하는 것"이 아니라
"공통 기준으로 쓰기 어려운 데이터를 저장 단계에서 차단하는 것"입니다.

나중에 서버 보정 저장이 필요해질 수는 있습니다.
다만 그때는 raw 값, canonical 값, 보정 근거, 원본 메타데이터까지 같이 설계해야 하므로
이번 작업과는 별도 단계로 보는 것이 맞습니다.

## 기대 결과

이 보강이 적용되면 기록 데이터 품질을 저장 시점에서 보장할 수 있습니다.

- 잘못된 기록이 분석, 티어, 랭킹까지 전파되는 일을 줄일 수 있습니다.
- 기록 기반 기능들의 계산 기준을 더 일관되게 유지할 수 있습니다.
- 이후 기록 집계와 품질 정책을 문서화하고 확장하기 쉬워집니다.

결국 이 작업의 목적은 validation을 늘리는 것이 아니라,
서비스 전체가 의존하는 기준 데이터를 신뢰할 수 있게 만드는 것입니다.

## 구현 후 확인할 항목

구현이 끝나면 아래 케이스를 기준으로 확인할 예정입니다.

- 정상 기록은 기존 계약을 유지한 채 저장되는지
- 시간 역전 기록이 저장되지 않는지
- `durationSec` 불일치 기록이 거부되는지
- `avgPaceSecPerKm` 불일치 기록이 거부되는지
- 비정상 범위 값이 거부되는지
- 실패 시 클라이언트가 처리 가능한 에러 응답이 내려가는지

## PR 리뷰 반영: 테스트 날짜 하드코딩 제거

기록 저장 검증 테스트에서 `2026-04-10` 같은 특정 날짜를 하드코딩한 부분이 있었습니다.
처음 의도는 유효한 과거 기록을 테스트하는 것이었습니다.

하지만 이 방식은 테스트 실행 날짜에 따라 의미가 달라집니다.

```text
테스트 실행일이 2026-04-13 이후:
2026-04-10은 과거라서 정상 테스트로 동작

테스트 실행일이 2026-04-10 이전:
2026-04-10은 미래라서 FUTURE_RECORD_NOT_ALLOWED 발생 가능
```

즉 테스트 데이터가 현재 시간에 의존하면서도 고정 날짜로 작성되어
실행 환경에 따라 실패할 수 있는 flaky test가 됩니다.

그래서 정상 기록, 시간 역전, duration 불일치, 평균 페이스 불일치,
허용 오차 경계 테스트는 현재 시각 기준 과거 시간을 사용하도록 바꿨습니다.

```java
private LocalDateTime validPastStartAt() {
    return LocalDateTime.now().minusMinutes(30);
}
```

반대로 미래 기록 거부 테스트는 검증 의도가 미래 기록 차단이므로
상대적인 미래 시간을 유지했습니다.

```java
LocalDateTime startAt = LocalDateTime.now().plusMinutes(6);
```

이 변경으로 테스트 의미가 아래처럼 분리되었습니다.

```text
정상/다른 검증 케이스:
항상 현재보다 과거인 기록 사용

미래 기록 검증 케이스:
항상 허용 범위를 넘는 미래 기록 사용
```

이 리뷰에서 얻은 기준은 아래와 같습니다.

```text
현재 시간 검증이 들어간 테스트에서는 절대 날짜를 고정하지 말고,
테스트 의도에 맞는 상대 시간을 사용한다.
```

## 후속 이슈: startAt/endAt 시간대 계약 정리

기록 저장 검증을 운영 환경에서 확인하는 과정에서
`startAt`, `endAt`이 실제 미래 시간이 아닌데도 `FUTURE_RECORD_NOT_ALLOWED`로 처리될 수 있는 문제가 발견되었습니다.

원인은 요청 시간이 `LocalDateTime` 형식이라 시간대 정보가 없다는 점입니다.

```json
{
  "startAt": "2026-04-14T12:13:09",
  "endAt": "2026-04-14T12:14:12"
}
```

위 값은 한국 시간처럼 보이지만,
문자열 자체에는 `+09:00`이나 `Z` 같은 offset 정보가 없습니다.
서버가 UTC 기준 환경에서 동작하면 한국 시간 기준 과거 기록도 미래 기록처럼 오판할 수 있습니다.

이 문제는 기록 검증 로직을 없애서 해결할 문제가 아니라,
API 계약에 시간대 정보를 포함하도록 바꾸는 것이 맞습니다.

후속 문서:

```text
docs/archive/records/running-record-time-contract-original.md
```

후속 이슈의 핵심은 아래입니다.

```text
POST /records 요청의 startAt/endAt을 OffsetDateTime 계약으로 전환한다.
```
