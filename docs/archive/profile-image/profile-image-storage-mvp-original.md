# STAR 정리: 프로필 이미지 저장 방식을 MVP 기준으로 어떻게 단순화했는가

- Status: Archived
- Audience: Engineers, Codex
- Source of Truth: No
- Last Reviewed: 2026-06-20

## Situation

우리 서비스에서 이미지가 쓰이는 지점은 아직 많지 않다.
현재 기준으로는 아래 두 종류가 핵심이다.

- 티어별 기본 이미지
- 사용자가 직접 설정하는 프로필 이미지

이 상황에서 처음부터 Cloudflare Images, R2, S3 같은 별도 스토리지를 붙이는 것은
운영 복잡도와 비용 측면에서 과하다고 판단했다.

반대로 프론트 번들에 티어 이미지를 넣는 방식도 검토했지만,
티어 이미지가 바뀌거나 티어 등급별 이미지가 추가될 때마다 앱 업데이트가 따라와야 한다는 점이 걸렸다.

즉 이 작업의 핵심은 "이미지 사용량은 적지만, 나중에 확장 가능한 기준은 남겨두면서
지금은 비용과 복잡도를 최소화하는 방식"을 정하는 것이었다.

## Task

정리해야 했던 질문은 아래 다섯 가지였다.

1. 티어 기본 이미지는 어디에서 관리할 것인가
2. 사용자 커스텀 이미지는 어디에 저장할 것인가
3. DB에는 무엇을 저장할 것인가
4. 프론트에는 어떤 형태의 값을 응답할 것인가
5. `customProfileImage`, `image_type`, `image_name` 같은 분리 구조가 지금 정말 필요한가

## Action

### 1. 티어 기본 이미지는 백엔드 정적 리소스로 관리하기로 했다

티어 기본 이미지는 사용자가 수정하는 데이터가 아니라
백엔드가 통제하는 메타데이터에 가깝다.

그래서 아래 방식으로 정리했다.

- 실제 파일은 백엔드 서버 디스크의 정적 리소스로 관리
- `tier` 테이블에는 전체 URL이 아니라 상대 경로만 저장
- 응답을 만들 때 필요한 경우 절대 URL로 조립

이 방식을 택한 이유는 다음과 같다.

- 같은 티어 사용자에게 같은 기본 이미지를 일관되게 보여줄 수 있다
- 이미지 교체가 필요할 때 앱 업데이트 없이 백엔드 데이터와 리소스만 바꾸면 된다
- 변경 이력을 서버 기준으로 관리하기 쉽다

### 2. 커스텀 프로필 이미지는 MVP에서 도커 볼륨 기반 저장을 전제로 했다

사용자 이미지까지 바로 외부 스토리지로 보내는 대신,
MVP 단계에서는 서버가 파일을 받아 디스크에 저장하고
컨테이너 재배포 시 파일이 사라지지 않도록 볼륨을 사용하는 방향을 잡았다.

기준 플로우는 아래와 같다.

1. 앱이 갤러리 또는 카메라에서 이미지를 선택한다
2. 앱이 서버로 업로드 요청을 보낸다
3. 서버가 파일을 검사한 뒤 디스크에 저장한다
4. DB에는 바이너리가 아니라 이미지 경로 또는 URL만 저장한다
5. 프로필 조회 시 앱은 그 값을 받아 화면에 표시한다

여기서 볼륨을 전제로 한 이유는 단순하다.
컨테이너 내부 경로에만 저장하면 재배포나 컨테이너 교체 시 파일이 함께 사라질 수 있기 때문이다.

즉 현재 판단은 "단일 서버 MVP에서는 볼륨 기반 로컬 저장으로 충분하지만,
서버가 여러 대가 되거나 스토리지 요구사항이 커지면 그때 오브젝트 스토리지로 옮긴다"에 가깝다.

### 3. 업로드와 교체 규칙은 커스텀 이미지만 대상으로 잡았다

사용자 이미지 업로드 시 필요한 처리 기준도 같이 정리했다.

- 파일이 실제 이미지인지 검사
- 용량 제한 검사
- 확장자 검사
- 저장 파일명 새로 생성
- 디스크 저장
- 기존 사용자 이미지가 있으면 교체 처리
- DB 갱신
- 최종 URL 반환

특히 파일명은 원본 이름을 그대로 쓰지 않고 새로 생성해야 한다고 봤다.

- 파일명 충돌 방지
- 한글, 공백, 특수문자 처리 단순화
- 원본 파일명 노출 방지
- 확장자 처리 일관성 확보

이미지 재변경 시 기준도 아래처럼 잡았다.

1. 기존 이미지가 커스텀 이미지인지 확인한다
2. 새 이미지를 저장한다
3. DB를 새 경로로 갱신한다
4. 예전 커스텀 이미지를 삭제한다

여기서 중요한 규칙은 하나다.

- 기본 이미지는 삭제 대상이 아니다
- 커스텀 이미지만 삭제 대상이다

### 4. API 응답은 표시 목적의 `displayProfileImage` 하나만 두기로 했다

처음에는 `displayProfileImage`와 `customProfileImage`를 나누는 방식도 생각했다.
하지만 프론트 입장에서 실제로 필요한 값은
"지금 화면에 어떤 이미지를 보여줄 것인가" 하나뿐이었다.

그래서 응답은 아래처럼 단순화하는 쪽이 더 낫다고 판단했다.

- `displayProfileImage`
  - 커스텀 이미지가 있으면 그 URL
  - 없으면 현재 티어 기본 이미지 URL

이렇게 하면 프론트는 분기 없이 이 값만 `img src`에 사용하면 된다.
커스텀 여부 판단은 백엔드 내부 규칙으로만 관리하면 된다.

### 5. `image_type`, `image_name` 컬럼 분리는 지금 도입하지 않기로 했다

겉으로 보면 `user.image_type`, `user.image_name`처럼 나누는 구조가 직관적으로 보일 수 있다.
하지만 현재 요구사항에서는 오히려 데이터 의미가 더 애매해질 수 있다고 봤다.

예를 들어 아래 질문이 바로 생긴다.

- `DEFAULT`일 때 `image_name`은 티어 코드인지 파일명인지 전체 경로인지
- 현재 티어가 바뀌면 user 테이블에 저장된 기본 이미지 정보는 어떻게 동기화할지
- 표시용 계산값과 저장용 원천값이 user 테이블에 같이 섞이지 않는지

그래서 지금 단계에서는 더 단순한 모델을 유지하기로 했다.

- 기본 이미지는 `tier` 메타데이터에서 관리
- 커스텀 이미지는 `user.profile_image` 같은 저장 필드 하나로 관리
- 응답에서는 최종 표시용 값만 `displayProfileImage`로 반환

## Result

최종적으로 MVP 기준 방향은 아래처럼 정리됐다.

- 티어 기본 이미지는 백엔드 정적 리소스로 관리한다
- 티어 테이블에는 기본 이미지의 상대 경로만 저장한다
- 커스텀 프로필 이미지는 볼륨 기반 로컬 저장을 전제로 한다
- DB에는 이미지 바이너리가 아니라 다시 요청 가능한 경로 또는 URL만 저장한다
- 프론트에는 `displayProfileImage` 하나만 반환한다
- 기본 이미지와 커스텀 이미지의 구분 책임은 백엔드 내부에 둔다
- `image_type`, `image_name`, `customProfileImage` 같은 추가 분리는 지금 도입하지 않는다

이렇게 정리하면 현재 서비스 규모에서 오버스펙을 피하면서도,
나중에 스토리지 전략을 바꿔야 할 때 응답 계약과 데이터 모델을 크게 흔들지 않을 수 있다.

## 이 기준이 유효한 범위

이 판단은 아래 전제에서 유효하다.

- 이미지 종류가 많지 않다
- 사용자 수와 트래픽이 아직 크지 않다
- 서버가 단일 인스턴스이거나, 업로드 디렉터리를 persistent volume으로 유지할 수 있다

반대로 아래 상황이 오면 다음 단계를 다시 검토해야 한다.

- 서버가 여러 대로 늘어난다
- 재배포 외에도 서버 교체가 잦아진다
- 썸네일, 리사이징, CDN, 접근 제어가 필요해진다
- 이미지 트래픽이 커져 정적 서빙 부담이 커진다

그 시점에는 Cloudflare Images/R2, S3 같은 외부 스토리지로 옮기는 것이 더 자연스럽다.
