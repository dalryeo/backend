# Git Commit Convention

- Status: Active
- Audience: Engineers, Codex
- Source of Truth: Yes
- Last Reviewed: 2026-06-15

## 현재 기준

달려 백엔드는 제목과 본문이 있는 작은 커밋을 사용한다.

- 커밋 제목은 `type: 제목` 형식을 쓴다.
- `type`은 소문자 영어로 쓴다.
- 제목은 한국어로 쓰고 명사형에 가깝게 끝낸다.
- 본문은 작성한다.
- 한 커밋에는 한 가지 논리적 변경만 담는다.
- `git add .` 대신 작업한 파일만 명시해서 stage한다.

## 제목 형식

```text
type: 제목
```

제목 규칙:

- 제목은 무엇이 바뀌는지 바로 드러나게 쓴다.
- 제목 끝에 마침표를 찍지 않는다.
- `한다`, `수정한다`, `추가한다` 같은 서술형 표현으로 끝내지 않는다.
- 파일명 나열보다 변경 목적을 쓴다.
- `이것저것 수정`, `전체 구조 개선`, `버그 수정`처럼 범위가 흐린 제목을 쓰지 않는다.

좋은 예시:

- `docs: 문서 하네스 구조 정리`
- `feat: 온보딩 응답 프로필 이미지 필드 분리`
- `refactor: 주간 티어 계산 로직 공통 resolver 추출`
- `fix: 기록 요약 응답 null 처리 보완`
- `test: 온보딩 서비스 프로필 이미지 테스트 추가`

피해야 할 예시:

- `docs: README.md 수정`
- `feat: 온보딩 응답 프로필 이미지 필드를 분리한다`
- `refactor: 전체 구조 개선`
- `fix: 버그 수정`

## type 선택 기준

주된 변경 의도를 기준으로 type을 고른다.

| type | 기준 |
| --- | --- |
| `feat` | 사용자 기능, API 계약, 응답 필드, 정적 리소스처럼 제품 동작이 바뀌는 변경 |
| `fix` | 버그 수정, 잘못된 분기 처리, 예외 처리 보완, 운영 장애 수정 |
| `refactor` | 동작은 유지하면서 구조를 개선하는 변경 |
| `test` | 테스트 코드 추가, 테스트 보강, 테스트 환경 정비 |
| `docs` | 문서만 수정하는 변경 |
| `chore` | 빌드, 설정, CI/CD, 의존성, 운영성 관련 변경 |
| `perf` | 성능 개선이 핵심인 변경 |

여러 종류의 변경이 섞이면 커밋을 나누는 것을 먼저 검토한다. 나누기 어렵다면 사용자에게 확인한다.

## 본문 작성 규칙

본문은 제목 아래 한 줄을 비우고 작성한다.

- 핵심 변경 내용을 1개 이상 적는다.
- 가능하면 1~3개 불릿으로 줄인다.
- 무엇이 바뀌었는지와 왜 필요한지를 중심으로 쓴다.
- 테스트, 배포, 롤백 메모가 있으면 함께 적는다.
- 구현 세부사항을 모두 나열하지 않는다.

예시:

```text
docs: 문서 하네스 구조 정리

- docs README와 indexes 추가
- 문서 구조 검증 스크립트 추가
- 기존 문서는 파일 단위 승인 후 이관하도록 명시
```

Codex가 명령으로 커밋할 때는 본문을 두 번째 `-m` 이후에 적는다.

```bash
git commit -m "docs: 문서 하네스 구조 정리" \
  -m "- docs README와 indexes 추가" \
  -m "- 문서 구조 검증 스크립트 추가"
```

## 스테이징 규칙

- 작업한 파일만 명시적으로 stage한다.
- unrelated 파일은 함께 stage하지 않는다.
- `.DS_Store`, 빌드 산출물, 로컬 설정 파일은 stage하지 않는다.
- 문서 이관처럼 파일 이동이 있는 작업은 이동 전후 경로를 확인한다.

좋은 예시:

```bash
git add AGENTS.md docs/README.md docs/indexes/standards-index.md scripts/docs-lint.sh
```

피해야 할 예시:

```bash
git add .
git add -A
```

## 커밋 전 확인

커밋 전에는 변경 범위에 맞는 검증을 실행한다.

- 문서 구조 변경: `bash scripts/docs-lint.sh`
- 코드 동작 변경: `./scripts/test-local.sh` 또는 관련 테스트
- 특정 테스트만 필요한 경우: `./scripts/test-local.sh --tests <테스트 클래스명>`

검증을 실행하지 못했다면 커밋 본문이나 작업 보고에 그 이유를 적는다.
